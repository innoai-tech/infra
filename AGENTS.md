# AGENTS

## 作用域

本文件作用于仓库 root；若下层目录新增更近的 `AGENTS.md`，则由下层覆盖其显式重写部分。

## 规则优先级

1. 用户当前明确请求。
2. 更近目录的 `AGENTS.md`。
3. 本 root `AGENTS.md`。

只有落盘到 `AGENTS.md` 的规则才视为仓库协作规则。对话中的临时偏好或即兴说明，未写入 `AGENTS.md` 前不视为持久规则。

## 变更边界

1. 开始改动前先做最小盘点，明确本次涉及的文件与执行入口。
2. 只修改与当前任务直接相关的仓库内文件。
3. 不因目录已存在而默认新增模块、入口、注册关系、workspace 边界或执行面。
4. 不回滚、覆盖或删除自己未引入的变更，除非用户明确要求。
5. 保持控制面文件边界清晰：协作规则写在 `AGENTS.md`，概述与索引写在 `README.md`，执行入口写在 `justfile`。

## 暂停条件

遇到以下情况先暂停并确认：

1. 需求边界不清，且存在多种互斥实现。
2. 需要新增模块、命令入口、注册关系、workspace 边界或执行面模式。
3. 请求与仓库当前结构、分层或长期维护约束明显冲突。
4. 关键依赖、运行环境或外部系统无法在当前工作区内完成验证。

## 人工接管条件

遇到以下情况应交由用户决策：

1. 需要执行破坏性操作。
2. 本地已有变更与当前任务直接冲突。
3. 仓库结构异常明显，局部 patch 会掩盖真实问题。

## 基本工程约束

1. 能做最小相关验证时，先验证再交付。
2. 无法验证时，必须明确说明缺口、原因与风险。
3. 优先通过参数传递或依赖注入传递上下文，避免可变全局状态。
4. 保持既有架构分层，不做与任务无关的顺手重构。
5. 交付说明只陈述已完成且已验证的事实，假设项与风险项要显式区分。
6. 避免过度工程、过早抽象、无必要注释与批量风格清理。
7. 尽量使用中文，除非专用名称，包括但不限于文档、注释、错误和日志信息。
8. 当用户请求与代码正确性、架构约束或长期维护成本明显冲突时，先指出风险，再讨论可接受折中。
